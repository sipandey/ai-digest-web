First Run vs Subsequent Runs

  • First run: user_delivered_papers is empty → all N papers from the cache are "fresh" → no cross-day filtering
  • Subsequent runs: M previously-delivered arXiv IDs are loaded and removed from the pool before shortlisting → user never sees the same paper twice
  • Deduplication is per-user — User A's delivery history does not affect User B's paper pool
  • Within a single day, re-running replays from the same papers_cache snapshot; only the specific papers the user hasn't received yet are re-evaluated

Shared vs Per-User Data

  • papers_cache — one row per arXiv paper per calendar day; written once, read by all users that day
  • paper_rankings_cache — per user × per paper × per profile hash × per prompt version; if the user edits their topics, old cached scores are ignored and new ones are generated
  • user_delivered_papers — lifetime ledger per user; never cleared; grows every successful run
  • pipeline_runs — one row per user per calendar day; tracks status and metrics for the dashboard

Shortlisting Pipeline (per user)

  • Step 1 — arXiv keyword filter: shared; drops papers not matching any of 5 keyword groups (RAG, Agents, LLMs, Multimodal, Safety)
  • Step 2 — Cross-day dedup: per-user; removes papers the user already received on past days
  • Step 3 — Topic shortlist: per-user; Python keyword overlap (title ×3, group ×2, abstract ×1) against the user's topic list; keeps top 40
  • Step 4 — LLM scoring: per-user; GPT-4o-mini scores each paper 1–10 on 4 rubric criteria; only papers ≥ 7.0 pass
  • Step 5 — LLM summarisation: per-user; GPT-4o-mini writes 5 structured fields for each passing paper

Scheduled vs Manual Runs

  • Scheduled (cron): check job runs every hour for ~15 s; heavy pipeline job only starts when users are actually due; uses OpenAI Batch API (50% cost, ~minutes latency)
  • Manual (Run Now): skips the time-of-day filter → all active users processed; uses synchronous OpenAI calls for immediate results; capped at 3 triggers/user/day with 5-min post-success cooldown
  • Both paths share the same papers_cache — a manual run at noon never re-fetches arXiv if the scheduled run already ran that morning
  • Concurrent manual triggers are race-safe: only one wins the DB atomic update to pending; the other gets a 400 "already running"

Multi-User Isolation

  • Each user runs inside an independent try/except block — one user's Notion error or OpenAI timeout never stops other users
  • Scoring and summarisation use per-user profile_hash as a cache key so two users with identical topics but different experience levels get different scores
  • Notion delivery is per-user (each user's own database and integration token)
  • The GitHub Actions concurrency group allows one scheduled run + unlimited simultaneous manual dispatch jobs

Key Thresholds & Limits

  • arXiv fetch: 5 categories × 150 results = up to 750 raw papers/day (de-duped across categories)
  • Keyword filter: only papers matching at least one of 5 groups pass
  • Shortlist cap: MAX_SHORTLIST = 40 papers sent to the LLM per user
  • Score threshold: SCORE_THRESHOLD = 7.0 / 10
  • Abstract truncation: 400 chars for scoring; full abstract for summarisation
  • Notion block limit: 100 blocks/append call; long digests chunked automatically
